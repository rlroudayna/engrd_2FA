# Requirements Document

## Introduction

This feature adds Two-Factor Authentication (2FA) to an existing Node.js/Express web application. 2FA requires users to verify their identity using a second factor (a time-based one-time password) in addition to their password. The goal is to reduce the risk of account compromise from stolen or guessed passwords.

## Glossary

- **System**: The Node.js/Express web application backend
- **User**: An authenticated account holder of the web application
- **2FA**: Two-Factor Authentication — a security mechanism requiring two distinct verification steps
- **TOTP**: Time-Based One-Time Password — a 6-digit code generated by an authenticator app (e.g. Google Authenticator, Authy) per RFC 6238
- **TOTP_Secret**: A unique base32-encoded secret key generated per user, used to derive TOTP codes
- **Authenticator_App**: A third-party mobile application that generates TOTP codes (e.g. Google Authenticator, Authy)
- **QR_Code**: A scannable image encoding the TOTP provisioning URI, used to register the TOTP_Secret in an Authenticator_App
- **Recovery_Code**: A single-use alphanumeric code that allows a User to bypass 2FA when access to their Authenticator_App is lost
- **Session**: An authenticated server-side session established after successful login
- **Enrollment**: The process by which a User registers a TOTP_Secret and activates 2FA on their account
- **Partial_Session**: A temporary server-side session state indicating the User has passed password verification but has not yet completed 2FA verification

## Requirements

### Requirement 1: 2FA Enrollment

**User Story:** As a User, I want to enroll in 2FA, so that I can secure my account with a second verification factor.

#### Acceptance Criteria

1. WHEN a User requests 2FA enrollment, THE System SHALL generate a unique TOTP_Secret for that User
2. WHEN a TOTP_Secret is generated, THE System SHALL return a QR_Code encoding the TOTP provisioning URI in the format `otpauth://totp/{issuer}:{username}?secret={secret}&issuer={issuer}`
3. WHEN a User submits a valid TOTP code during enrollment, THE System SHALL store the TOTP_Secret and mark the User's account as 2FA-enabled
4. WHEN a User submits an invalid TOTP code during enrollment, THE System SHALL reject the enrollment and return a descriptive error message
5. WHILE a User's 2FA enrollment is pending confirmation, THE System SHALL NOT mark the account as 2FA-enabled
6. WHEN 2FA enrollment succeeds, THE System SHALL generate exactly 8 Recovery_Codes for the User and return them once for the User to store
7. THE System SHALL store Recovery_Codes as hashed values using a one-way hashing algorithm

### Requirement 2: 2FA Verification at Login

**User Story:** As a User with 2FA enabled, I want to be prompted for my TOTP code after entering my password, so that my account is protected even if my password is compromised.

#### Acceptance Criteria

1. WHEN a User with 2FA enabled submits valid credentials, THE System SHALL establish a Partial_Session and require TOTP verification before granting full access
2. WHEN a User with 2FA disabled submits valid credentials, THE System SHALL establish a full Session without requiring TOTP verification
3. WHEN a User submits a valid TOTP code during login, THE System SHALL replace the Partial_Session with a full Session
4. WHEN a User submits an invalid TOTP code during login, THE System SHALL reject the request and return a descriptive error message
5. WHEN a User submits an invalid TOTP code 5 consecutive times within a single Partial_Session, THE System SHALL invalidate the Partial_Session and require the User to restart the login process
6. THE System SHALL accept TOTP codes within a 30-second time window, allowing one step of clock drift (±30 seconds)

### Requirement 3: Recovery Code Usage

**User Story:** As a User who has lost access to my Authenticator_App, I want to use a Recovery_Code to log in, so that I can regain access to my account.

#### Acceptance Criteria

1. WHEN a User submits a valid unused Recovery_Code during 2FA verification, THE System SHALL grant a full Session and mark that Recovery_Code as used
2. WHEN a User submits an already-used Recovery_Code, THE System SHALL reject the request and return a descriptive error message
3. WHEN a User submits an invalid Recovery_Code, THE System SHALL reject the request and return a descriptive error message
4. WHEN a User has fewer than 3 remaining Recovery_Codes, THE System SHALL include a warning in the login response indicating the User should regenerate Recovery_Codes
5. WHEN a User requests regeneration of Recovery_Codes, THE System SHALL invalidate all existing Recovery_Codes and generate a new set of exactly 8 Recovery_Codes

### Requirement 4: 2FA Disabling

**User Story:** As a User, I want to disable 2FA on my account, so that I can remove the second factor if needed.

#### Acceptance Criteria

1. WHEN a User requests to disable 2FA, THE System SHALL require the User to provide their current password and a valid TOTP code before disabling
2. WHEN the provided password and TOTP code are both valid, THE System SHALL remove the TOTP_Secret, invalidate all Recovery_Codes, and mark the account as 2FA-disabled
3. IF the provided password or TOTP code is invalid, THEN THE System SHALL reject the disable request and return a descriptive error message

### Requirement 5: TOTP Secret Security

**User Story:** As a system operator, I want TOTP secrets stored securely, so that a database breach does not immediately compromise all 2FA-protected accounts.

#### Acceptance Criteria

1. THE System SHALL encrypt TOTP_Secrets at rest using AES-256 encryption before storing them in the database
2. THE System SHALL never include the TOTP_Secret in any API response after enrollment is complete
3. WHEN a User's account is deleted, THE System SHALL delete the associated TOTP_Secret and all Recovery_Codes

### Requirement 6: API Endpoints

**User Story:** As a frontend developer, I want well-defined API endpoints for all 2FA operations, so that I can integrate 2FA into the user interface.

#### Acceptance Criteria

1. THE System SHALL expose a `POST /auth/2fa/enroll` endpoint that initiates 2FA enrollment and returns a QR_Code
2. THE System SHALL expose a `POST /auth/2fa/enroll/verify` endpoint that confirms enrollment with a TOTP code
3. THE System SHALL expose a `POST /auth/2fa/verify` endpoint that accepts a TOTP code or Recovery_Code during login
4. THE System SHALL expose a `POST /auth/2fa/disable` endpoint that disables 2FA for the authenticated User
5. THE System SHALL expose a `POST /auth/2fa/recovery-codes/regenerate` endpoint that regenerates Recovery_Codes for the authenticated User
6. WHEN any 2FA endpoint receives a request from an unauthenticated User, THE System SHALL return an HTTP 401 response
7. THE System SHALL return all API responses in JSON format
